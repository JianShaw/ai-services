# 未来知识库治理能力说明

## 背景

当前知识库 RAG 链路已经具备基础能力：

- 上传知识文档
- 文本切块
- 写入 Qdrant 向量库
- 聊天时按知识库检索
- 基于检索结果生成回答
- 通过 `status=active` 控制可检索知识范围

这个阶段解决的是“能不能查到并回答”的问题。

相关存储结构说明见：

- [知识库存储模型说明](./2026-05-11-knowledge-storage-model.md)

随着知识文档变多、业务规则变化、客服策略变复杂，知识库还需要进一步具备“可运营、可治理”的能力。核心是两个方向：

1. 更细粒度的知识分类
2. 版本、生效时间与优先级控制

这两个能力不是当前 MVP 必须项，但会影响后续客服系统是否能长期稳定运营。

---

## 一、chunk 级分类的意义

### 当前状态

目前 `category` 主要是文档级别字段。例如一整篇文档可能都属于：

```text
faq
```

但一篇电商客服知识文档内部可能同时包含：

```text
工作时间
退货政策
退款政策
物流说明
发票规则
优惠券规则
投诉处理
```

如果这些内容都只归类为 `faq`，系统就只能知道“这是一篇常见问题文档”，但不知道每个 chunk 具体属于哪类业务知识。

### 业务问题

当知识库扩大后，文档级分类会遇到这些问题：

- 后台无法按“退货政策”“物流说明”“优惠券规则”等业务类型精细管理知识。
- 检索时无法优先限定某类知识，可能把相邻但不相关的内容一起召回。
- 不同业务类型无法走不同回答策略，例如投诉、维权、隐私、支付异常类问题应更保守。
- 运营侧无法统计用户最常问的是哪类知识。
- 更新某一类知识时，难以判断影响范围。

### 目标能力

未来应支持 chunk 级分类：

```text
工作时间 -> working_hours
七天无理由退货 -> return_policy
退款到账 -> refund_policy
发货与物流 -> shipping_policy
发票规则 -> invoice_policy
会员与优惠券 -> coupon_policy
投诉处理 -> complaint_policy
```

这样系统可以做到：

- 用户问退货问题时，优先检索 `return_policy`。
- 用户问退款到账时，优先检索 `refund_policy`。
- 用户问投诉、维权、监管时，进入更保守策略或直接转人工。
- 后台可以按业务分类统计、筛选、编辑知识。

### 设计方向

文档仍然保留默认分类：

```text
KnowledgeDocument.category = faq
```

但 chunk 可以拥有自己的分类：

```text
KnowledgeChunk.meta_data.category = return_policy
```

写入 Qdrant 时优先使用 chunk 分类：

```text
Qdrant metadata.category = chunk.category or document.category
```

## 二、版本、生效时间与优先级的意义

### 当前状态

当前文档已有 `version` 字段，但检索时没有使用版本、生效时间或优先级逻辑。

在文档较少时，这不是问题。但当同一类规则多次更新后，可能出现旧规则和新规则同时处于 active 状态。

### 业务问题

例如旧规则是：

```text
退款 1-3 个工作日到账
```

新规则变成：

```text
退款 3-7 个工作日到账
```

如果旧文档没有失效，新文档也被上传为 active，RAG 可能同时召回两种说法，最终导致 AI 回答不一致或答错。

类似场景还包括：

- 大促期间物流时效临时变化。
- 春节期间客服工作时间调整。
- 某规则从指定日期开始生效。
- 不同租户、店铺、渠道使用不同政策。
- 旧政策需要保留归档，但不能再被 AI 引用。

### 目标能力

未来知识库需要能判断：

- 哪条知识当前有效？
- 哪条知识已经过期？
- 多条知识都匹配时，哪条优先？
- 旧版本是否还能被检索？

可选字段：

```text
effective_from
effective_to
priority
version
status
```

推荐状态：

```text
indexing
active
index_failed
archived
disabled
expired
```

### 设计方向

短期可以采用简单版本策略：

```text
同一个 tenant_id + category 只允许一个 active 文档
```

上传新版本成功后，将同租户、同分类下的旧 active 文档改为：

```text
archived
```

并删除旧文档对应的 Qdrant 向量。

中长期再加入：

```text
effective_from_ts
effective_to_ts
priority
```

检索时只召回：

```text
status = active
effective_from <= now
effective_to is null or effective_to > now
```

如果多条知识同时有效，再按 `priority` 或版本排序。

## 三、建议实施阶段

### 阶段 1：保持现状，稳定基础链路

目标：

- 上传稳定
- Qdrant 入库稳定
- active/status 过滤稳定
- 删除和禁用不会残留脏向量
- 检索日志可观测

适用当前 MVP 阶段。

### 阶段 2：支持 chunk 级分类

触发时机：

- 知识文档超过 10 份
- 后台需要按业务类型管理知识
- 不同业务类型需要不同回答策略
- 需要统计高频知识类型

主要改动：

- 切块时根据标题识别 chunk 分类
- chunk metadata 写入 `category`
- Qdrant metadata 使用 chunk 分类
- 后台展示 chunk 分类

### 阶段 3：支持简单版本治理

触发时机：

- 同一类政策开始频繁更新
- 出现新旧规则同时存在的风险
- 运营需要保留旧文档但禁止 AI 使用

主要改动：

- 新文档成功入库后，自动归档同租户、同分类旧 active 文档
- 检索只查 active 文档
- 后台展示 archived/active 状态

### 阶段 4：支持生效时间和优先级

触发时机：

- 活动期规则和平时规则不同
- 政策需要提前配置，到点自动生效
- 多租户、多渠道规则开始分化

主要改动：

- 增加 `effective_from_ts`
- 增加 `effective_to_ts`
- 增加 `priority`
- 检索时按时间窗口和优先级过滤

## 四、判断是否需要开发的信号

### 应该开发 chunk 级分类的信号

- 后台运营开始问“能不能只看退货政策？”
- 用户问题统计需要区分退货、退款、物流、发票、优惠券。
- 投诉、维权、隐私等问题需要特殊处理。
- 检索结果经常混入相邻但不相关的知识。

### 应该开发版本治理的信号

- 同一政策存在多个版本。
- 旧规则导致 AI 答错。
- 活动期规则需要自动生效或失效。
- 客服答案必须严格遵守当前有效政策。

---

## 五、一句话总结

chunk 级分类解决的是：

```text
这条知识属于什么业务问题？
```

版本、生效时间和优先级解决的是：

```text
这条知识现在是否还能用？如果多条都能用，哪条优先？
```

这两类能力的目的不是让当前 demo 回答得更好，而是为后续知识库规模扩大、业务规则变化、客服策略分流提供治理基础。
