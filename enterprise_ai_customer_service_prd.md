# 企业级 AI 客服系统 PRD

## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 产品名称 | 企业级 AI 客服系统 |
| 版本 | v0.1 / MVP |
| 面向用户 | 电商 / SaaS / 物流 / 本地生活 / 售后服务型企业 |
| 产品定位 | 一个可接入企业知识库、订单系统和工单系统的 AI 客服 Agent，用于自动接待、问题解答、业务查询、工单流转和人工协同 |
| 核心目标 | 降低人工客服压力，提高响应速度，提升问题解决率，并沉淀客服知识与运营数据 |

---

## 2. 背景与问题

传统客服系统主要依赖人工客服和固定规则机器人，常见问题包括：

1. 人工客服成本高，高峰期响应慢。
2. 传统 FAQ 机器人只能命中固定关键词，无法理解复杂表达。
3. 企业知识分散在文档、后台系统、飞书/钉钉/Notion、订单系统、工单系统中，客服查询效率低。
4. 客服新人培训成本高，回答质量不稳定。
5. 用户问题无法被系统化沉淀，后续知识库优化困难。
6. 多渠道客服割裂，例如官网、微信、App、企微、抖音、小程序、邮件等渠道数据不统一。

因此需要一个具备“理解问题、查询知识、调用业务系统、判断风险、转人工、沉淀数据”能力的企业级 AI 客服系统。

---

## 3. 产品目标

### 3.1 业务目标

1. 自动回答常见问题，降低人工客服咨询量。
2. 支持订单、物流、退款、售后等业务查询。
3. 对复杂问题自动创建工单并分配给人工客服。
4. 提升客服响应速度和 7×24 小时服务能力。
5. 建立客服知识沉淀和持续优化机制。

### 3.2 技术目标

1. 支持 RAG 知识库问答。
2. 支持工具调用，例如订单查询、物流查询、退款进度查询。
3. 支持多轮上下文对话。
4. 支持意图识别、任务路由、转人工、风控拦截。
5. 支持会话日志、调用链追踪、质量评估和运营看板。
6. 支持多渠道接入。

### 3.3 MVP 成功指标

| 指标 | 目标 |
|---|---|
| 首响时间 | AI 首次响应 < 3 秒 |
| 常见问题自动解决率 | ≥ 40% |
| 人工转接准确率 | ≥ 85% |
| 知识库命中率 | ≥ 60% |
| 用户满意度 | ≥ 80% |
| 高风险问题误答率 | 尽量接近 0，必须有兜底策略 |

## 4. 用户角色

### 4.1 终端用户 / 客户

企业的真实客户，例如买家、会员、商户、企业用户。

核心诉求：

1. 快速得到准确答案。
2. 查询订单、物流、退款、售后进度。
3. 遇到复杂问题时能快速转人工。
4. 不想反复描述问题。

### 4.2 人工客服

负责处理 AI 无法解决的问题。

核心诉求：

1. 快速看到用户问题摘要。
2. 看到 AI 已经问过什么、查过什么、回答过什么。
3. 快速接管会话。
4. 可以一键采用 AI 推荐回复。
5. 可以标记错误回答，反哺知识库。

### 4.3 客服主管 / 运营人员

负责管理客服质量和知识库。

核心诉求：

1. 查看 AI 解决率、转人工率、满意度。
2. 查看高频问题和未解决问题。
3. 管理知识库内容。
4. 审核 AI 回答质量。
5. 配置不同业务场景下的回复策略。

### 4.4 企业管理员

负责系统配置、权限、渠道、模型和数据安全。

核心诉求：

1. 配置渠道接入。
2. 配置员工权限。
3. 配置模型、知识库、工具、敏感词和安全策略。
4. 查看系统运行状态和成本。

## 5. 核心业务场景

### 5.1 FAQ 问答

用户询问通用问题，例如：

- 发货时间是多久？
- 如何申请退款？
- 保修期多久？
- 发票怎么开？
- 会员权益有哪些？

AI 应基于知识库回答，并尽量引用知识来源。

### 5.2 订单查询

用户询问：

- 我的订单到哪了？
- 订单为什么还没发货？
- 退款什么时候到账？
- 能不能改地址？

AI 需要识别用户身份，调用订单系统查询，并根据状态生成回复。

### 5.3 售后处理

用户反馈：

- 商品损坏。
- 收到错货。
- 服务无法使用。
- 需要退换货。

AI 需要判断是否可以自助处理，必要时收集必要信息并创建工单。

### 5.4 投诉与高风险问题

用户表达强烈不满、威胁投诉、法律纠纷、敏感信息泄露、支付异常等。

AI 不应随意承诺赔偿或做出高风险判断，应转人工或进入审核流程。

### 5.5 人工协同

当 AI 无法解决、置信度低、用户要求人工、涉及敏感业务时，应自动转人工。

转人工时需要带上：

1. 用户基本信息。
2. 会话摘要。
3. 用户意图。
4. 已查询的业务数据。
5. AI 已尝试的解决方案。
6. 推荐客服回复。

## 7. 功能需求

# 7.1 用户端聊天窗口

## 功能描述

用户可以通过 Web 页面与 AI 客服进行对话。

## 功能点

1. 展示欢迎语。
2. 支持文本输入。
3. 支持快捷问题按钮。

用户：我的订单怎么还没发货？

用户：是的。

系统调用订单查询工具。

AI：您的订单 202604300001 当前状态为“待出库”，预计今天 18:00 前发货。如超过预计时间仍未发货，我可以帮您创建催发货工单。

# 7.3 意图识别与路由

## 功能描述

系统需要识别用户问题属于哪个业务场景，并路由到对应处理流程。

## 意图类型

| 意图 | 说明 | 处理方式 |
|---|---|---|
| faq | 通用 FAQ 问题 | RAG 知识库回答 |
| order_query | 订单查询 | 调用订单工具 |
| logistics_query | 物流查询 | 调用物流工具 |
| refund_query | 退款查询 | 调用退款工具 |
| after_sales | 售后问题 | 收集信息 / 创建工单 |
| complaint | 投诉 | 优先转人工 |
| human_request | 用户要求人工 | 转人工 |
| unknown | 无法识别 | 追问 / 转人工 |

## 路由原则

1. 高置信度 FAQ：直接进入知识库问答。
2. 订单类问题：必须调用业务系统，不允许纯编造。
3. 售后类问题：先收集必要信息，再创建工单。
4. 投诉 / 法律 / 赔偿 / 隐私 / 支付风险：转人工。
5. 低置信度问题：追问一次，仍不明确则转人工。

# 7.4 知识库问答

## 功能描述

基于企业知识库进行 RAG 问答。

## 知识来源

1. FAQ 文档。
2. 产品说明文档。
3. 售后政策。
4. 物流政策。
5. 退款政策。
6. 内部客服话术。
7. 常见问题沉淀。

## 功能点

1. 文档上传。
2. 文档解析。
3. 文本切片。
4. 向量化。
5. 相似度检索。
6. rerank，可在第二阶段加入。
7. 答案生成。
8. 来源引用。
9. 无答案兜底。
10. 知识库版本管理。

## 回答规则

1. 只能基于召回知识回答。
2. 不确定时必须说明无法确认。
3. 不得编造政策。
4. 涉及金额、赔付、法律、隐私问题时走人工。
5. 回复应简洁、清楚、礼貌。

# 7.5 工具调用能力

## 功能描述

AI 可以调用企业内部系统工具完成业务查询。

## MVP 工具列表

| 工具 | 说明 |
|---|---|
| get_user_profile | 查询用户信息 |
| get_recent_orders | 查询用户最近订单 |
| get_order_detail | 查询订单详情 |
| get_logistics | 查询物流信息 |
| get_refund_status | 查询退款状态 |
| create_ticket | 创建工单 |
| transfer_to_human | 转人工 |

## 工具调用原则

1. 订单、物流、退款等实时信息必须调用工具。
2. 工具调用失败时要返回可理解的错误提示。
3. 不能把内部错误直接暴露给用户。
4. 工具调用需要日志追踪。
5. 涉及权限的数据必须鉴权。

# 7.6 转人工

## 触发条件

1. 用户明确说“转人工”“找客服”“人工”。
2. AI 连续两轮无法解决。
3. 意图识别置信度低。
4. 涉及投诉、法律、赔偿、隐私、支付风险。
5. 工具调用失败且无法继续处理。
6. 用户评价“不满意”。
7. 超出 AI 权限范围。

## 转人工信息包

转人工时系统应自动生成：

```json
{
  "session_id": "s_001",
  "user_id": "u_001",
  "intent": "refund_query",
  "summary": "用户询问退款进度，AI 已查询订单和退款状态，退款仍在处理中。用户表示不满。",
  "tools_used": ["get_order_detail", "get_refund_status"],
  "risk_level": "medium",
  "suggested_reply": "您好，我看到您的退款还在处理中，我帮您进一步核实原因。"
}
```

## 8. Agent 流程设计

## 8.1 总体流程

```text
用户输入
  ↓
安全检查 / 敏感识别
  ↓
意图识别 Router
  ↓
┌───────────────┬───────────────┬───────────────┐
↓               ↓               ↓
FAQ             订单查询          售后 / 投诉
↓               ↓               ↓
RAG 检索         工具调用          信息收集 / 转人工
↓               ↓               ↓
生成回复         生成业务回复       创建工单
  ↓
质量检查
  ↓
回复用户 / 转人工
  ↓
记录日志与反馈
```

## 8.2 LangGraph 节点建议

| 节点 | 作用 |
|---|---|
| input_guard_node | 输入安全检查 |
| intent_router_node | 意图识别与路由 |
| faq_rag_node | 知识库检索 |
| order_tool_node | 订单工具调用 |
| logistics_tool_node | 物流工具调用 |
| refund_tool_node | 退款工具调用 |
| after_sales_node | 售后信息收集 |
| ticket_node | 工单创建 |
| human_transfer_node | 转人工 |
| answer_generate_node | 生成回复 |
| answer_review_node | 回复质量检查 |
| memory_node | 保存会话上下文 |
| analytics_node | 写入统计日志 |

## 8.3 add_edge 与 route 使用建议

固定流程使用 add_edge：

```text
输入安全检查 → 意图识别 → 回复质量检查 → 保存日志
```

动态决策使用 route：

```text
意图识别后，根据 intent 路由到 FAQ / 订单 / 售后 / 转人工
```

质量检查后也使用 route：

```text
如果回答可信 → 回复用户
如果回答不可信 → 转人工或重新检索
```

## 9. 非功能需求

# 9.1 性能要求

| 项目 | 要求 |
|---|---|
| AI 首响 | < 3 秒 |
| 工具查询 | < 2 秒 |
| RAG 检索 | < 1 秒 |
| 会话页面加载 | < 2 秒 |
| 并发能力 | MVP 支持 100 并发，后续扩展到 1000+ |

# 9.2 稳定性要求

1. 模型调用失败时自动降级。
2. 工具调用失败时有兜底回复。
3. RAG 无结果时不能编造。
4. 转人工能力必须可用。
5. 会话记录不能丢失。

# 9.3 安全要求

1. 用户数据必须鉴权。
2. 内部工具不能被用户 prompt injection 任意调用。
3. 敏感字段脱敏展示。
4. 日志中避免保存明文密码、身份证、银行卡等敏感信息。
5. 高风险业务需要人工确认。
6. 知识库文档需要权限隔离。

# 9.4 可观测性要求

1. 每次会话有 trace_id。
2. 每次 LLM 调用记录 prompt 版本、模型、token、耗时、结果。
3. 每次工具调用记录入参、出参、耗时、错误。
4. 每次路由记录 intent、confidence、decision。
5. 支持查看完整调用链。

## 10. 数据模型初稿

# 10.1 会话表 conversations

```sql
CREATE TABLE conversations (
  id VARCHAR(64) PRIMARY KEY,
  user_id VARCHAR(64),
  channel VARCHAR(32),
  status VARCHAR(32),
  current_intent VARCHAR(64),
  summary TEXT,
  assigned_agent_id VARCHAR(64),
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);
```

# 10.2 消息表 messages

```sql
CREATE TABLE messages (
  id VARCHAR(64) PRIMARY KEY,
  conversation_id VARCHAR(64),
  sender_type VARCHAR(32),
  content TEXT,
  metadata JSONB,
  created_at TIMESTAMP
);
```

# 10.3 工单表 tickets

```sql
CREATE TABLE tickets (
  id VARCHAR(64) PRIMARY KEY,
  conversation_id VARCHAR(64),
  user_id VARCHAR(64),
  type VARCHAR(64),
  priority VARCHAR(32),
  status VARCHAR(32),
  description TEXT,
  assigned_to VARCHAR(64),
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);
```

# 10.4 知识库文档表 knowledge_documents

```sql
CREATE TABLE knowledge_documents (
  id VARCHAR(64) PRIMARY KEY,
  title VARCHAR(255),
  source_type VARCHAR(64),
  status VARCHAR(32),
  version VARCHAR(32),
```

## 17. 关键业务规则

1. 订单、物流、退款问题必须调用工具，不允许模型直接猜。
2. AI 不允许承诺赔偿金额。
3. AI 不允许修改订单、退款、地址等高风险操作，MVP 只允许查询和创建工单。
4. 用户明确要求人工时必须转人工。
5. 知识库无命中时不能编造答案。
6. 连续两次低置信度回答后必须转人工。
7. 所有 AI 回复必须记录来源、模型、耗时和 trace_id。
8. 所有工具调用必须记录成功 / 失败。

## 18. 验收标准

## 18.1 FAQ 问答验收

给定 50 个标准 FAQ 问题：

1. AI 能正确回答 ≥ 80%。
2. 无答案问题不会编造。
3. 敏感问题能正确转人工。

## 18.2 订单查询验收

给定 20 个订单问题：

1. AI 能正确调用订单工具。
2. AI 能根据订单状态生成用户可理解回复。
3. 工具失败时能给出兜底提示。

## 18.3 转人工验收

给定 20 个转人工场景：

1. 用户明确要求人工时 100% 转人工。
2. 投诉 / 法律 / 赔偿类问题 100% 转人工。
3. 转人工时带完整会话摘要。

## 18.4 工单验收

1. AI 能根据售后问题创建工单。
2. 工单能关联会话。
3. 客服能在工作台查看工单。

## 21. MVP 最小可做版本总结

第一版不要追求“大而全”，重点做这条主线：

```text
用户提问
  ↓
AI 判断意图
  ↓
FAQ 走知识库
订单问题走 Mock Tool
复杂问题转人工 / 创建工单
  ↓
保存会话日志
  ↓
后台可查看和复盘
```

只要这条链路跑通，就已经不是普通 ChatBot，而是一个具备企业级雏形的 AI 客服 Agent 系统。
